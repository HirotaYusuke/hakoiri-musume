# 体験差別化プラン（案出し段階）

**本文書は案出し段階のメモであり、確定したスコープや承認済みの仕様ではない。** 採用する方向性や優先順位は別途検討・決定が必要である。

---

## 目的とスコープ

- **目的**: ゲームの差別化を、**ユーザー体験（UX）**の側面から整理し、後続の実装・改修の優先検討材料とする。
- **本チャット／本プランの主な対象**:
  - UI（見た目、操作感、フィードバックの層構造）
  - 効果音・オーディオ全般の方針
  - ゲーム外・メタ機能（フック、記録、ストア文脈など）
  - アクセシビリティ（a11y）に関する方針・訴求の整理
- **明示的に対象外とするもの**:
  - **パズルの中身**（問題内容、レベル設計、ステージロジックなど）。これらは別途・別タスクで扱う前提とする。

---

## 決定ログ（確定事項）

以下は内部での確定事項として記録する。本文書の案出しセクションと矛盾する場合は、原則として本セクションを優先する。

### 2026-05-13

- **アートディレクション**は一本化する。単一の統一した世界観およびルールに揃える。
- **手数・自己ベスト**については、クリア後に手数を表示する。自己ベスト（最小手数）には**サーバー／データベースを使わない**。永続化する場合はブラウザの **`localStorage`** を前提とし、キーは**問題 ID 単位**などとする。データは端末ローカルのみに留め、**端末間の同期や改ざん耐性は狙わない**。
- **次の設計ブロック**として、一本化したアートに沿った **UI ガイドライン**（色・質感・タイポグラフィ・余白・コンポーネントの角や影など）をまとめる。**効果音**も同じ世界観に揃える。実装順としては **アート／UI → 音** の順が自然である。

## 差別化の軸（他ゲーム比較）

- **認知負荷を抑えつつ奥行きを持たせる**: ルールの複雑さで勝負せず、**理解コストは低く**、操作とフィードバックの積み重ねで「深さ」を感じさせる。
- **スライド操作と脱出の触感**: 駒の**スライド**と**壁・段との接触**、**脱出（クリア）時の手応え**を、視覚・音・微細な動きで一貫して伝える。
- **短時間プレイと広告配信への適合**: セッションが短く切れやすい構造と親和性のある体験設計（待たせない、区切りが明確、繰り返し遊んでも疲れにくいフィードバック設計を志向）。
- **「ルールの差」ではなく「手触りと記憶」で差別化**: プレイ後に残るのは手順の暗記より、**感触・リズム・空間のイメージ**であることを意識する。

## UI方向性

- **レンガ（ブロック）としての一貫した質感**:
  - **重み**と**わずかな影**で「おもちゃの積み木」ではなく、**建築的・物質的な塊**として読めるビジュアルを目指す。
- **脱出表現は「穴」ではなく物語性のある隙間**:
  - 単なる抜け穴ではなく、**隙間・光・奥行き**など、クリア地点が「抜けるべき場所」として感情的・空間的に伝わる見せ方を検討する。
- **駒の識別はテキストに依存しない**:
  - **色**を主な識別子としつつ、**色覚多様性**に配慮するため**微妙なテクスチャやパターン**で補完する（色だけに依存しない）。
- **フィードバックのレイヤー（煩さを抑える）**:
  - **無効手**: 控えめなシェイク＋こもった／弱めの音など、**主張はするが騒がしくない**組み合わせ。
  - **有効手**: **短いスライド**と**わずかな慣性**で「動いた実感」を出し、長いアニメや派手すぎる演出は避ける。
- **ワンタップの静かモード**:
  - PWA／モバイル利用を想定し、**効果音を素早くオフ（または極小）にできる**入口を1回の操作で用意する方針とする。

## 効果音方向性

- **Web Audio との互換性**を前提に、実装・設計の制約内で破綻しない短いワンショット中心とする。
- **質感**: レンガ同士のこすれ・軽い打音に近い**短いタップ／擦過系**を基本とし、**ローパスなどで高域の安っぽさ**を抑え、安価な効果音寄りの印象を避ける。
- **イベント分離**（例）:
  - **移動成功**、**壁ヒット**、**ライン／エリアクリア**など、**意味の違う操作には別の短音**を割り当て、判別しやすくする。
- **尺**: ベースは概ね**80〜150ms程度**の短いものを想定し、**ループや長いテール**は避け、オーバーラップ時もむしろ重くならないよう設計する。

## ゲーム外・メタ

すべて任意案であり、実装の有無は別途判断する。

- **デイリー／ウィークリー1問**のような、**毎日・毎週1パズルだけ触れるフック**の可能性。
- **クリア後の手数表示**と**自己ベスト（手数）**の比較。**ランキング前提は不要**とし、個人の改善実感を主とする。**サーバー／DB による記録やランキングは対象外**であり、永続化の方針は**決定ログ（2026-05-13）**を参照する（必要ならブラウザの `localStorage` と問題 ID 単位のキーなど）。
- **ストア・説明文向けのアクセシビリティストーリー**: 色に加え**形・パターン補助**があること、関連する設定や配慮を**短く明確に**伝えられる素材の準備を検討する。

## エージェント実行時の前提

- **パズルカタログ／ゲームエンジン本体の仕様変更は、別途タスクで明示されない限りスコープ外**とする。体験レイヤー（UI・音・メタ・a11y訴求）に集中する。
- **小さな差分で積み上げる**: 可能な限り**PR単位でレビューしやすい粒度**の変更に分割する。
- **検証**: 既存テストを壊さないことを前提に、**視覚・聴覚の確認**を実施する（自動テストで取りきれない部分は手動確認の必要性を明記する）。
